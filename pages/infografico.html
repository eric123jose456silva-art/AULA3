🔨 Anatomia de RF (OBRIGATÓRIA)

Todo requisito profissional segue este padrão:

[Ator] + [Dever] + [Ação] + [Objeto] + [Condição]
Exemplo Correto:
O cliente + deve + validar + CPF + em < 1s
⚠️ Sem um destes 5 elementos = RF ruim!

🔗 Rastreabilidade (ESSENCIAL)

Cada RF deve ter uma RN (Regra de Negócio) pai:

RN.01.001: Aumentar receita
↓
RF.01.001: Validar pagamento
↓
Código: validarPagamento()
🚨 RF SEM RN PAI = ÓRFÃO
Ninguém pediu! Feature Creep!
🔴 Sinais de Alerta (RF Ruim)
  • "deve ser rápido" ← Vago!
  • "melhorar UX" ← Genérico!
  • "integração BD" ← Sem ator!
  • "gerenciar vendas" ← Grande demais!
  • Sem RN pai ← ÓRFÃO!
  • Teste ambíguo ← Talvez? Não!
🟢 Sinais de Ouro (RF Bom)
  • ✅ Ator identificado
  • ✅ Verbo específico (validar, calcular, enviar)
  • ✅ Métrica/Condição testável
  • ✅ Ligado a RN pai
  • ✅ Resultado binário: Passou ou Falhou
✨ Se RF tem estes 5 = PROFISSIONAL
🏆 5 Critérios de Ouro (VRNCR)
V - Verificável Pode ser testado? Existe forma de medir?
R - Rastreável Pode ser ligado a uma regra de negócio?
N - Não-ambíguo Todos entendem igual?
C - Completo Tem ator, dever, ação, objeto, condição?
R - Realizável É tecnicamente possível?
✅ Checklist de Qualidade
  • Cada RF tem um ator identificado?
  • Nenhum RF usa palavras vagas?
  • Cada RF é testável por QA?
  • Todo RF tem uma RN pai?
  • MoSCoW foi consensuado?
  • RFs categorizados D/P/I/S?
  • Conflitos foram identificados?
  • Pelo menos 5 RFs MUST?

≥ 70% = APROVADO ✓

❌ vs ✅ Exemplos Reais

❌ Ruim: "O sistema deve processar transações rápido"

✅ Bom: "O sistema deve processar transação em < 500ms com 99.9% uptime"

❌ Ruim: "Melhorar experiência do usuário"

✅ Bom: "Cliente completa login em máx 3 cliques com tempo ≤ 2 segundos"

❌ Ruim: "Integração com banco de dados"

✅ Bom: "Sistema sincroniza dados com BD em cada transação com confirmação ACID"
