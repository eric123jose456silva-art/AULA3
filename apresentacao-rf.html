1. A Transição Crítica: Do Porquê ao O Quê

A Jornada do Pensamento Estratégico

Regra de Negócio (RN): Responde ao PORQUÊ - O objetivo estratégico

Requisito Funcional (RF): Responde ao O QUÊ - A função que realiza

RN → RF → CÓDIGO
Visão Estratégica → Especificação Técnica → Implementação

⚠️ Erro Comum: Confundir um requisito técnico com a razão pelo qual ele existe

2

Exemplos Comparativos: RN vs RF

🏦 SISTEMA BANCÁRIO

RN: "Minimizar fraudes e garantir conformidade regulatória"

RF: "O sistema deve validar transações contra regras de detecção de anomalias em tempo real"

🛒 E-COMMERCE

RN: "Aumentar conversão reduzindo atrito no checkout"

RF: "O sistema deve autofill de endereço a partir do CEP com validação IBGE"

🏥 APP DE SAÚDE

RN: "Garantir continuidade do cuidado médico"

RF: "O sistema deve sincronizar histórico médico entre dispositivos do paciente"

2. Anatomia do Requisito de Ouro

A Estrutura Técnica Obrigatória

Um requisito profissional e testável segue esta sintaxe:

[Ator] + [Dever de Sistema] + [Ação] + [Objeto] + [Condição de Sucesso]
Exemplo: O usuário cliente deve poder consultar seu extrato bancário com precisão de 99.9% em menos de 2 segundos.
4

Decomposição da Anatomia

Elemento por Elemento

Elemento Função Exemplo
Ator Quem interage com o sistema Cliente, Gerente, Sistema Externo
Dever de Sistema Obrigatoriedade do sistema deve, pode, não deve
Ação Verbo específico (testável) consultar, calcular, gerar, validar
Objeto O que é manipulado extrato, saldo, relatório
Condição de Sucesso Critério de aceitação testável em tempo real, com 99.9% precisão
5

Tabela de Transformação: Antes e Depois

Convertendo Requisitos Amadores em Profissionais

❌ Requisito Inadequado ✅ Requisito Profissional
"O sistema deve ser rápido" "O sistema deve processar consultas em ≤ 500ms"
"Melhorar a experiência do usuário" "O usuário deve completar login em máx 3 cliques"
"Integração com banco de dados" "O sistema deve sincronizar dados com BD em cada transação com confirmação ACID"
"Gerar relatórios úteis" "O gestor deve exportar relatório de vendas em PDF com gráficos em até 5s"
"Segurança no sistema" "O sistema deve criptografar senhas com bcrypt + salt aleatório de 12 rounds"
6

Os 5 Critérios de Ouro de um RF

Como Identificar um Requisito de Qualidade

  • 1. Verificável: Pode ser testado? Existe forma de medir?
  • 2. Rastreável: Pode ser ligado a uma regra de negócio?
  • 3. Não-ambíguo: Todos entendem igual?
  • 4. Completo: Ator, ação, objeto e condição presentes?
  • 5. Realizável: É tecnicamente possível dentro de prazo/orçamento?
Pergunta Chave: Se mostro este requisito para 5 desenvolvedores diferentes, eles escrevem o mesmo código?

3. Categorização de Requisitos Funcionais

Tipo 1: Requisitos de Dados

Definem o que o sistema armazena e como acessa

RF.D.001: O sistema deve armazenar dados de clientes (nome, CPF, endereço, telefone) em tabela relacional normalizada.
RF.D.002: O sistema deve permitir busca de produtos por nome, categoria, preço com resposta em ≤ 1 segundo.

Inclui: Estrutura de dados, validações, integridade, constraints, índices para performance.

8

Categorização (continuação)

Tipo 2: Requisitos de Processamento

Lógica de negócio - Como o sistema transforma dados

RF.P.001: O sistema deve calcular automaticamente o frete baseado em CEP, peso e dimensão da caixa, usando tabela SEDEX 2024.

Tipo 3: Requisitos de Interface (Lógica)

Interação usuário-sistema - Como o usuário dispara ações

RF.I.001: O cliente deve visualizar carrinho atualizado em tempo real conforme adiciona/remove produtos, com total de preço recalculado.
9

Categorização (conclusão)

Tipo 4: Requisitos de Saída (Relatórios)

Dados entregues ao usuário - Formato, frequência, conteúdo

RF.S.001: O gestor deve receber relatório diário de vendas em PDF via email às 8h, contendo: total de vendas, ticket médio, produtos top-5, gráfico de tendência.

Resumo das 4 Categorias

Tipo Foco Pergunta-Chave
Dados (D) O quê armazenar Quais dados e estruturas?
Processamento (P) Como transformar Qual a lógica?
Interface (I) Como interagir Como o usuário usa?
Saída (S) Como entregar Que informação sai?

4. Metodologia MoSCoW na Prática

Muito Além da Sigla

MoSCoW não é apenas um acrônimo. É uma estratégia de decisão que responde:

"Com recurso X, em prazo Y, qual requisito sacrificamos primeiro?"

M - MUST (Deve ter) Sem isso, o projeto fracassa. É a viabilidade básica. Impacto: 🔴🔴🔴
S - SHOULD (Deveria ter) Muito importante, mas pode esperar iteração futura. Impacto: 🟠🟠
C - COULD (Poderia ter) Nice-to-have. Se sobrar tempo/dinheiro. Impacto: 🟡
W - WON'T (Não terá) Explicitamente descartado nesta iteração. Impacto: ⚪
11

Impacto Financeiro e de Prazo

Como Cada Categoria Afeta o Projeto

MoSCoW Impacto de Prazo Impacto Financeiro Risco
MUST Crítico - atraso afeta tudo Orçamento base + contingência Alto: cancelamento se não entregar
SHOULD Importante - pode impactar ROI +20-40% do orçamento base Médio: satisfação cliente reduz
COULD Flexível - ajusta com folga de prazo +5-15%, geralmente descartado Baixo: polimento apenas
WON'T Sem impacto nesta iteração Zero neste ciclo Nenhum: planejado para futuro

⚠️ Regra de Ouro: MUST deve ser 30-40% do escopo. Se tiver mais, seu projeto está em risco.

12

Como um Analista Decide?

A Árvore de Decisão MoSCoW

1. Viabilidade Crítica?
SIM → MUST | NÃO → próxima pergunta
2. ROI ou Conformidade?
SIM → SHOULD | NÃO → próxima pergunta
3. Tempo disponível?
SIM → COULD | NÃO → WON'T
4. Valor Real vs Esforço?
Valor > Esforço × 2 → considerar SHOULD
Exemplo - App de Saúde:
• Login seguro = MUST (sem isso, falha)
• Histórico completo = SHOULD (importante mas não inviabiliza)
• Tema customizável = COULD (bom ter)
• Integração com wearable = WON'T (próxima versão)
13

Cenário de Crise: Prazo Reduzido

Como Sacrificar Requisitos Inteligentemente

Situação Real: Projeto de 6 meses, prazo reduzido para 3. O que fazer?

Estratégia de Corte

  1. Primeiro corte: Todos os WON'T automático → ganho 0% (já não eram planos)
  2. Segundo corte: 70% dos COULD → ganho ~10-15% de tempo
  3. Terceiro corte: 40% dos SHOULD → ganho ~15-20% de tempo (risco: 20% redução ROI)
  4. Último recurso: Renegociar MUST ou splittar em MVP + v1.1
Nunca Sacrifique MUST
Se precisar cortar MUST, você não tem um projeto viável. Hora de replanejar escopo ou prazo.

5. Rastreabilidade Obrigatória

Por Que Todo RF Precisa de uma RN Pai

Rastreabilidade é a corrente que liga Negócio → Requisito → Código

RN01 → Aumentar receita em 20%
↓
RF01 → Criar recomendação de produtos
↓
Código → Algoritmo machine learning
Benefícios:
✓ Saber POR QUÊ cada requisito existe
✓ Identificar requisitos órfãos (feature creep)
✓ Validar mudanças: "se mudar RF01, muda RN01?"
✓ Facilitar cortes: "WON'T RF03 → atrasado? Cancela sem culpa"
15

Prática: Numeração e Rastreamento

Sistema Profissional de Identificação

Formato Recomendado

RN[Módulo][Sequência] → RF[Módulo][Sequência]
Exemplo: RN.01.02 → RF.01.02 (Módulo 1, Requisito 2)
ID Descrição Rastreabilidade Status
RN.01.01 Reduzir fraude bancária — Ativa
RF.01.01 Validar transação em IA ← RN.01.01 Desenvolvimento
RF.01.02 Bloquear transação suspeita ← RN.01.01 Teste
RF.01.99 Dashboard de analytics ❌ ÓRFÃO Cancelada

⚠️ RF sem RN = Feature Creep (requisito que ninguém pediu)

Uma matriz que mostra a cobertura de cada requisito de negócio:

6. Oficina de Laboratório (Atividade Prática)

Roteiro Passo a Passo - Parte 1

🎯 Objetivo

Transformar RNs superficiais em RFs profissionais, categorizados, priorizados e rastreáveis.

📋 Entregáveis Esperados

  1. Planilha Excel com 15-20 RFs específicos do seu projeto
  2. Cada RF seguindo a estrutura: [Ator] + [Dever] + [Ação] + [Objeto] + [Condição]
  3. Categorização: D (Dados), P (Processamento), I (Interface), S (Saída)
  4. Priorização MoSCoW com justificativa
  5. Rastreabilidade 1:N com RNs

Tempo: 2 horas de trabalho em grupo (máx 4 pessoas)

Oficina de Laboratório - Parte 3

Checklist de Qualidade Antes de Entregar

  • ☐ Cada RF tem um ator identificado (pessoa ou sistema)?
  • ☐ Nenhum RF usa palavras vagas ("rápido", "bom", "melhor")?
  • ☐ Cada RF pode ser testado por um QA ou automatizado?
  • ☐ Todo RF tem uma RN pai justificando sua existência?
  • ☐ A priorização MoSCoW foi consenso do grupo?
  • ☐ RFs foram categorizados em D / P / I / S?
  • ☐ Requisitos conflitantes foram identificados e anotados?
  • ☐ Pelo menos 5 RFs estão marcados como MUST?

📊 Critério de Aprovação

Mínimo 70% dos requisitos devem passar em todos os 8 critérios acima
